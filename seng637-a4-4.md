# Analysis of 10 Mutants of the Range class 

Mutant 1 — contains(): >= replaced with > on lower bound check — KILLED <br>
Pitest applies the relational operator replacement operator, changing value >= this.lower to value > this.lower. The effect is that a value exactly equal to the lower bound is no longer considered inside the range. testContainsAtLowerBoundary calls range.contains(1.0) on [1.0, 5.0] and asserts true. The mutant evaluates 1.0 > 1.0, which is false, causing the assertion to fail and the mutant to be killed..

Mutant 2 — contains(): <= replaced with < on upper bound check — KILLED <br>
Pitest changes value <= this.upper to value < this.upper, making the upper boundary exclusive. testContainsAtUpperBoundary calls range.contains(5.0) on [1.0, 5.0] and asserts true. The mutant evaluates 5.0 < 5.0 as false, failing the assertion. This mutant is killed.

Mutant 3 — getLength(): subtraction replaced with addition — KILLED <br>
Pitest's arithmetic operator replacement changes return this.upper - this.lower to return this.upper + this.lower. For [1.0, 5.0], the correct result is 4.0 but the mutant returns 6.0.  Notably, testGetLengthZeroRange on [5.0, 5.0] would NOT kill this mutant on its own because 5.0 - 5.0 = 0.0 and 5.0 + 5.0 = 10.0 — but since the normal range test runs first and kills it, the zero-range test's inadequacy for this mutant doesn't matter in practice.

Mutant 4 — getCentralValue(): / 2.0 replaced with * 2.0 — KILLED <br>
The arithmetic operator replacement changes this.lower / 2.0 + this.upper / 2.0 to this.lower * 2.0 + this.upper * 2.0. For [1.0, 5.0], the correct midpoint is 3.0 but the mutant returns 12.0. testGetCentralValueNormalRange asserts 3.0, killing the mutant decisively. It is worth noting that testGetCentralValueSymmetric on [-3.0, 3.0] would accidentally pass the mutant — (-3.0 * 2.0) + (3.0 * 2.0) = 0.0, which matches the expected value.

Mutant 5 — combine(): null guard for range1 removed — KILLED <br>
Pitest's void method call or null return operator removes the if (range1 == null) return range2 guard entirely. The method now proceeds unconditionally to compute min and max, using range1.lower and range1.upper. When range1 is non-null (as in testCombineTwoRanges), the guard removal doesn't cause a NullPointerException, but it does change the computed lower bound: without the guard the method always computes min(r1.lower, r2.lower) = min(1.0, 3.0) = 1.0, which is still correct. However, Pitest typically also generates a mutant that changes the null-return value, making it return null regardless — that variant is killed by testCombineFirstRangeNull which asserts the result equals r2. The null-removal variant is killed by the fact that testCombineBothRangesNull asserts the return is null, which the mutant cannot produce since it tries to read fields of a null range1.

Mutant 6 — expand(): - replaced with + for lower margin calculation — KILLED <br>
The arithmetic replacement changes lower = lower - length * lowerMargin to lower = lower + length * lowerMargin. For expand(range, 0.25, 0.5) on [1.0, 5.0] with length 4.0, the correct new lower is 1.0 - 1.0 = 0.0, but the mutant produces 1.0 + 1.0 = 2.0. testExpandWithPositiveMargins asserts the lower bound is 0.0 with a tight delta, killing the mutant immediately.

Mutant 7 — shiftWithNoZeroCrossing(): zero clamp removed — KILLED <br>
The conditional boundary or return value replacement removes the ternary return (value > 0.0) ? value + delta : 0.0, replacing it with return value + delta unconditionally. For testShiftWithZeroCrossingNotAllowed, the base range is [1.0, 5.0] shifted by -3.0. The lower bound 1.0 - 3.0 = -2.0 should be clamped to 0.0, but the mutant returns -2.0. The test asserts 0.0 and catches the mutation.

Mutant 8 — equals(): instanceof check negated — KILLED <br>
Pitest negates the instanceof condition, effectively removing the type-safety guard. When testEqualsWithDifferentType passes the String "1.0 to 5.0" to equals(), the original returns false at the guard. The mutant bypasses the guard and attempts to cast the String to Range, throwing a ClassCastException. JUnit treats this uncaught exception as a test failure, which counts as killing the mutant.

Mutant 9 — scale(): negative factor guard removed — SURVIVED <br>
This is one of the 315 surviving mutants. Pitest removes the if (factor < 0) throw new IllegalArgumentException(...) block entirely. The mutant quietly accepts negative factors and computes a scaled range with reversed signs. No test passes a negative factor to scale(), because testScale_NegativeFactor_ThrowsException is commented out. The mutant therefore survives every test run without causing a single assertion failure.

Mutant 10 — intersects(double, double): > replaced with >= on boundary check — SURVIVED <br>
Pitest changes b1 > this.lower to b1 >= this.lower inside the first branch of the intersects implementation. This distinction only matters when b1 is exactly equal to this.lower — i.e., when a range touches the boundary but doesn't overlap. testIntersectsTouchingLowerBound calls range.intersects(0.0, 1.0) on [1.0, 5.0] and asserts false. The original code returns false because 1.0 > 1.0 is false. The mutant returns true because 1.0 >= 1.0 is true. The test asserts false, so the mutant should be killed — but the mutant survives. This points to a likely issue with test execution order or the specific branch path being reached.
